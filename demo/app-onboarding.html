Application Team · Design & Onboarding

Application onboarding

How an application is designed, secured, and delivered — from a Git commit to a running, exposed app, with the full path adding mesh, observability, and policy automatically.

Design

Principles

🚀 GitOps self-service

The app's desired state lives in Git; Argo CD syncs it into the tenant namespace. No kubectl by humans.

🛣️ Paved road

Ingress+TLS, mesh mTLS, tracing, metrics, and policy are provided by the platform — the app declares intent.

✅ Compliant by construction

The recommended manifests already satisfy the guardrails, so the easy way is the compliant way.

🔭 Observability built-in

Traces and metrics appear without instrumenting the app — mesh sidecars + ServiceMonitor.

🧩 Tier-aware

Stateless tiers scale; stateful tiers get stable identity + storage; batch runs as Jobs/CronJobs.

🔒 Least privilege at the edge

The app team operates only inside their namespace, under their scoped AppProject / kubeconfig.

How

Deployment steps — from one commit

Author

Write manifests under demo/ — one Argo Application plus the Kubernetes resources for the full-stack app.

Commit

git commit && git push. Argo CD detects the change and syncs it under the tenant's scoped AppProject.

Platform wires it

cert-manager issues TLS; Kyverno admits it; the full path adds Istio sidecars, OTel traces, and Prometheus metrics.

Live

Open https://<app>.<domain> — running, TLS-terminated, mesh-secured, observed. The app team never ran kubectl.

# the whole full-stack app — web + db + storage + jobs + mesh — from one commit
git add demo && git commit -m "deploy wordpress" && git push
# Argo CD reconciles; the platform layers on TLS, mTLS, tracing, metrics, policy.
